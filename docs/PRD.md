# ER Scribe MVP PRD

## 🎯 핵심 정보

**목적**: 응급실 의사가 모바일로 문진 키워드를 빠르게 입력하면 AI가 병원 스타일의 차팅 초안 2종(HPI 줄글 + C.C 상용구)을 자동 생성해주는 어시스턴트
**사용자**: 응급실에서 모바일로 환자를 문진하고, 데스크톱으로 차트를 작성하는 응급의학과 의사

---

## 🚶 사용자 여정

서비스에는 두 가지 접근 경로가 있으며, 각각의 목적에 맞게 진입 화면이 다릅니다.

### 📱 모바일 PWA 경로 (문진 입력 목적)

```
1. 홈 화면의 앱 아이콘 탭 (설치형 PWA)
   ↓ 로그인 여부 확인

   [비로그인] → 로그인 페이지 → 인증 성공
   [로그인] ↓

2. 모바일 대시보드
   → 화면 상단: 큰 [새 케이스 시작] 버튼 (가장 눈에 띄는 CTA)
   → 화면 하단: 최근 케이스 3개 (스크롤)
   ↓ [새 케이스 시작] 버튼 탭

3. 새 케이스 입력 페이지 (전체화면, 모바일 최적화)
   ↓ 베드번호 선택 (구역 A/B/R 토글 → 해당 구역 번호 탭, 기본값 A구역)
   ↓ C.C 입력 (자동완성)
   ↓ 키워드 카드 입력 (반복)
   ↓ [차팅 생성] 버튼 탭

4. 케이스 결과 페이지로 자동 이동
   → 생성 중 상태 표시 (폴링)
   → 사용자는 앱을 닫고 이동해도 됨 (생성은 서버에서 계속)
```

### 🖥️ 데스크탑 웹 경로 (결과 확인 목적)

```
1. 브라우저에서 링크 접속
   ↓ 로그인 여부 확인

   [비로그인] → 로그인 페이지 → 인증 성공
   [로그인] ↓

2. 데스크탑 대시보드
   → 화면 중앙: 최근 케이스 목록 (그리드, 상태 배지 표시)
   → 우측 상단: [새 케이스] 버튼 (보조적 위치)
   ↓ 케이스 카드 클릭

3. 케이스 결과 페이지 (데스크탑 최적화)
   → [생성 중] → 폴링 → 자동 갱신
   → [완료] → HPI 줄글 + 상용구 차팅 표시
   → 편집 후 복사 → EMR 붙여넣기
```

## 📱 메뉴 구조

모바일 PWA와 데스크탑 웹은 **같은 URL 구조**를 사용하되, 레이아웃과 주요 CTA 배치가 다릅니다.

```
📱 ER Scribe 내비게이션

🌐 공개 메뉴 (비로그인, 공통)
├── 홈
└── 👤 인증
    ├── 로그인 - F010
    └── 회원가입 - F010

📱 모바일 PWA — 로그인 후
├── 🏠 대시보드 (진입 시 첫 화면)
│   ├── ⭐ [새 케이스 시작] — F001 (화면 상단, 대형 CTA)
│   └── 최근 케이스 3개 — F006 (하단 스크롤 영역)
├── 📋 케이스 목록 (보조 접근)
│   └── 기능: F006
├── 📖 가이드라인 관리
│   └── 기능: F012 (C.C별 커스텀 가이드라인 업로드·수정·삭제)
├── ⚙️ 설정
│   └── 기능: F013 (입력 화면 레이아웃 선택)
└── 🚪 로그아웃

🖥️ 데스크탑 웹 — 로그인 후
├── 🏠 대시보드 (진입 시 첫 화면)
│   ├── 케이스 목록 그리드 — F006 (화면 중앙, 주 영역)
│   └── [+ 새 케이스] 버튼 — F001 (우측 상단, 보조 위치)
├── 📋 케이스 목록 (전체 히스토리)
│   └── 기능: F006
├── 📖 가이드라인 관리
│   └── 기능: F012
├── ⚙️ 설정
│   └── 기능: F013
└── 🚪 로그아웃
```

> **PWA 진입 동작**: `manifest.json`의 `start_url`을 `/`로 설정. 홈 페이지에서 로그인 감지 시 `/dashboard`로 자동 이동. `/dashboard`는 뷰포트 크기 또는 PWA standalone 모드를 감지해 모바일/데스크탑 레이아웃을 각각 렌더링.

## 📌 주요 설계 결정 사항

1. **원본 텍스트 보존**: `case_inputs.raw_text`는 절대 수정하지 않음. AI 생성 JSON은 중간 표현이며, 원문이 진실의 원천
2. **재생성 히스토리**: `case_results`는 케이스당 다수(1:N). `cases.current_result_id`로 현재 표시본 지정. 이전 결과 유지로 재생성 후 비교 가능
3. **AI 파이프라인 분리**: 입력 정규화 → HPI 생성 → 상용구 생성을 3단계로 분리해 각 단계 독립 재생성 및 프롬프트 관리 가능
4. **structured_schema 확정 전까지 placeholder 진행**: 스키마는 실제 케이스 예시를 ChatGPT로 분석 후 확정 예정
5. **가이드라인 저장 분리**: 시스템 기본 가이드라인은 `lib/ai/resources/guides/{cc}.md` 파일로 Git 관리. 사용자 커스텀은 `interview_guidelines` 테이블에만 저장. C.C 입력 시 커스텀 있으면 커스텀 우선, 없으면 파일 읽어 표시
6. **레이아웃 설정 저장 위치**: 입력 화면 분할 방식은 `profiles.input_layout`에 저장. 별도 설정 테이블 없이 기존 profiles 확장으로 단순화
7. **베드번호는 구역/번호 2컬럼 분리**: 단일 텍스트(예: `A-3`) 대신 `bed_zone` enum + `bed_number` integer로 분리 저장. 이유 — (a) 구역별 CHECK 제약으로 잘못된 번호(예: R-10) 원천 차단, (b) 구역별 필터·정렬·집계 쿼리가 단순해짐, (c) UI에서 `${bed_zone}-${bed_number}` 조합으로 언제든 재구성 가능
8. **베드번호 UI는 2-탭 완료 원칙**: 구역 토글(기본 A) + 번호 버튼 그리드를 같은 화면 상단에 동시 렌더. 화면 전환이나 스크롤 없이 구역 → 번호 2번의 탭으로 선택 완료. C.C 입력보다 먼저 노출해 환자 추적용 메타데이터를 가장 먼저 확정
