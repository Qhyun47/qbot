# AI 차팅 아키텍처 설명 문서

> **대상 독자**: 이 프로젝트를 처음 보는 사람, 또는 시간이 지난 뒤 다시 읽는 사람. 개발 경험이 없어도 이해할 수 있도록 작성되었습니다.

---

## 1. 개요

규봇의 AI 차팅은 의사가 문진 중 입력한 짧은 메모(카드)들을 받아, 4가지 서식(HPI 줄글·P.I template·History·P/E)으로 이루어진 차팅 초안을 자동으로 생성합니다.

이 문서는 그 과정에서 **어떤 파일이 무슨 역할을 하는지**, **AI가 어떻게 판단하는지**, 그리고 **새로운 C.C.(주소 증상)나 예시를 추가할 때 어떤 순서로 진행해야 하는지**를 설명합니다.

AI 차팅 관련 파일들은 모두 Git(버전 관리 시스템)으로 관리됩니다. 데이터베이스(DB)에는 사용자가 직접 커스텀한 가이드라인만 저장하고, 나머지 AI 학습 자료는 모두 파일 형태로 프로젝트 안에 보관합니다.

---

## 2. AI 차팅 전체 흐름 — 5단계 쉬운 설명

의사가 [차팅 생성] 버튼을 누르면 서버에서 아래 5단계가 순서대로 실행됩니다.

```
[의사의 카드 입력]
       ↓
 1단계: 정규화
       ↓
 2단계: HPI 줄글 생성
       ↓
 3단계: P.I template 생성
       ↓
 4단계: History 생성
       ↓
 5단계: P/E 생성
       ↓
[결과 페이지에 표시]
```

### 1단계 — 정규화 (카드 내용을 구조화된 정보로 변환)

의사가 입력한 카드(예: "3일 전 흉통 시작, NTG 먹어도 안 나아짐")를 AI가 읽고, "이 정보는 증상 시작 시점이고, 저 정보는 약물 반응이다"처럼 항목별로 분류합니다.

이 단계의 결과물은 화면에 보이지 않고, 다음 단계들이 사용하는 중간 데이터로 쓰입니다.

참고 파일: `ai-docs/prompts/normalize.md`(AI에게 주는 지시문)

### 2단계 — HPI 줄글 생성 (현병력 서술)

정규화된 정보를 바탕으로 "상기 환자 3일 전부터 left chest의 squeezing pain이 시작되었으며…"와 같은 흐름 있는 문장을 만듭니다.

같은 C.C.의 모범 HPI 예시가 있으면(= 예시 파일에 등록된 경우) 그 스타일을 참고해 문체를 맞춥니다.

참고 파일: `ai-docs/prompts/generate-pi.md`

### 3단계 — P.I template 생성 (상용구 현병력 필드 채우기)

상용구(`template.json`)의 현병력 필드 구조에 맞춰 각 항목을 채웁니다. 예를 들어 "NTG response?" 항목에 "(-)"를 넣는 식입니다.

등록된 상용구가 없는 C.C.라면 이 단계는 빈 결과를 반환합니다.

참고 파일: `ai-docs/prompts/generate-template.md`, `ai-docs/templates/{templateKey}/template.json`

### 4단계 — History 생성 (과거력·복용약·수술력 작성)

Past Hx. / Med Hx. / Op Hx. (/ Family Hx.)를 작성합니다. 카드에 해당 정보가 없으면 `(-)`로 채웁니다. Family Hx.는 언급이 있을 때만 포함됩니다.

참고 파일: `ai-docs/prompts/generate-history.md`, `ai-docs/templates/{templateKey}/template.json` (history 섹션)

### 5단계 — P/E 생성 (신체 검진 소견 작성)

신체 검진 소견을 C.C. 상용구의 P/E 필드 구조에 맞춰 채웁니다.

참고 파일: `ai-docs/prompts/generate-pe.md`, `ai-docs/templates/{templateKey}/template.json` (pe 섹션)

## 3. 참조 파일 지도 — 어떤 파일이 무슨 역할인가

### AI에게 보내는 지시문 (프롬프트 파일)

| 파일 경로                              | 역할                                                                           |
| -------------------------------------- | ------------------------------------------------------------------------------ |
| `ai-docs/prompts/normalize.md`         | "카드 내용을 이렇게 분류해라" — 1단계 AI 지시문                                |
| `ai-docs/prompts/generate-pi.md`       | "HPI 줄글을 이렇게 써라" — 2단계 AI 지시문. 언어 규칙·시간 표기·문장 규칙 포함 |
| `ai-docs/prompts/generate-template.md` | "P.I template 필드를 이렇게 채워라" — 3단계 AI 지시문                          |
| `ai-docs/prompts/generate-history.md`  | "History를 이렇게 써라" — 4단계 AI 지시문                                      |
| `ai-docs/prompts/generate-pe.md`       | "P/E를 이렇게 써라" — 5단계 AI 지시문                                          |

### C.C.별 상용구 뼈대 (templateKey 폴더)

| 파일 경로                                       | 역할                                                                            |
| ----------------------------------------------- | ------------------------------------------------------------------------------- |
| `ai-docs/templates/{templateKey}/template.json` | 이 C.C.의 상용구 양식. `fields`(P.I template 필드)·`history`·`pe` 3개 섹션 포함 |
| `ai-docs/templates/{templateKey}/schema.json`   | 1단계 정규화 시 AI가 어떤 항목을 추출해야 하는지 알려주는 구조 정의             |
| `ai-docs/templates/{templateKey}/examples.md`   | 이 C.C.의 실제 케이스 예시 모음. AI가 스타일을 참고하는 학습 자료               |
| `ai-docs/templates/_generic/schema.json`        | 등록된 스키마가 없는 C.C.에 쓰이는 범용 스키마                                  |

### C.C.·가이드라인·상용구 목록

| 파일 경로                             | 역할                                                                                |
| ------------------------------------- | ----------------------------------------------------------------------------------- |
| `lib/ai/resources/cc-list.json`       | 등록된 모든 정형 C.C. 목록. 각 C.C.가 어떤 가이드라인·상용구와 연결되어 있는지 기록 |
| `lib/ai/resources/guide-list.json`    | 등록된 가이드라인 목록. 여기 없으면 가이드라인 드롭다운에 표시되지 않음             |
| `lib/ai/resources/template-list.json` | 등록된 상용구 목록. 여기 없으면 상용구 선택 화면에 표시되지 않음                    |

### 가이드라인 파일

| 파일 경로                              | 역할                                                       |
| -------------------------------------- | ---------------------------------------------------------- |
| `ai-docs/guides/{guideKey}/guide.html` | HWP에서 내보낸 가이드라인 HTML 파일. 문진 화면 패널에 표시 |

### 연결 대기·추적 파일

| 파일 경로                    | 역할                                                                       |
| ---------------------------- | -------------------------------------------------------------------------- |
| `ai-docs/pending-matches.md` | C.C./가이드라인/상용구 간 아직 연결되지 않은 항목을 임시로 기록하는 메모장 |

## 4. 상황별 참조 매핑

AI는 C.C.에 상용구가 있는지, 예시 파일이 있는지에 따라 다르게 동작합니다.

| 상황                        | AI가 참고하는 것                   | 결과                                                    |
| --------------------------- | ---------------------------------- | ------------------------------------------------------- |
| **상용구 있음 + 예시 있음** | 지시문 + 상용구 양식 + 예시 스타일 | 가장 완성도 높은 결과 생성                              |
| **상용구 있음 + 예시 없음** | 지시문 + 상용구 양식               | 양식은 채워지지만, 스타일 참고 없이 생성                |
| **상용구 없음 + 예시 있음** | 지시문 + 예시 스타일               | HPI만 생성. P.I template·History·P/E는 범용 스키마 사용 |
| **상용구 없음 + 예시 없음** | 지시문만                           | HPI만 생성. 범용 스키마 기반으로 기본 동작              |

> **요약**: 상용구를 등록하면 4개 섹션을 모두 생성할 수 있고, 예시를 추가할수록 실제 현장 스타일에 가까운 결과가 나옵니다.

## 5. 상용구 추가 절차

새로운 C.C.에 상용구를 추가할 때는 `/tmpl` 명령어를 사용합니다.

**전체 흐름 요약**

1. `/tmpl {templateKey}` 입력
2. 연결할 C.C. 지정 (있으면 바로 연결, 없으면 대기 목록에 기록)
3. P.I template 양식 제공 (현병력 구조화 필드와 출력 예시)
4. History 양식 제공
5. P/E 양식 제공
6. 케이스 예시 제공 (최소 1개, 많을수록 AI 품질 향상)
7. Claude가 프롬프트 파일에 새로운 패턴 반영 여부 분석 → 승인 시 적용

**각 단계가 필요한 이유**

- **양식 3종(P.I template·History·P/E)**: AI가 "이 상용구의 필드 구조가 이렇게 생겼구나"를 알아야 채울 수 있습니다.
- **케이스 예시**: 양식을 아는 것과 실제로 잘 채우는 것은 다릅니다. 실제 출력 예시를 주면 AI가 문체·표현·약어 스타일을 학습합니다.
- **메타 학습 분석**: 새 예시에서 기존 프롬프트 파일에 반영되지 않은 규칙이 발견되면 즉시 업데이트해서 다른 C.C.에도 품질이 전파됩니다.

## 6. 예시 추가 · 삭제 절차

### 예시 추가

기존에 이미 상용구가 등록된 C.C.에 예시만 추가할 때는 `/add-example {templateKey}` 명령어를 사용합니다.

**예시 파일 형식** (`ai-docs/templates/{templateKey}/examples.md`)

```
# Case 1
## HPI (줄글)
(AI가 생성한 또는 직접 작성한 HPI 줄글 예시)

## P.I template
(실제 출력 예시)

## History
(실제 출력 예시)

## P/E
(실제 출력 예시)

# Case 2
...
```

- 케이스를 추가하면 기존 파일 뒤에 이어 붙이고, 번호는 자동으로 증가합니다.
- 파일이 없으면 새로 만듭니다.

### 예시 삭제

예시 파일을 삭제하거나 상용구 자체를 삭제할 때는 아래 규칙을 따릅니다.

- **예시만 삭제**: `examples.md` 파일에서 해당 케이스 블록(`# Case N`부터 다음 `# Case` 전까지)을 직접 편집
- **상용구 전체 삭제**: `template-list.json`에서 제거 + `cc-list.json`의 `templateKeys`에서 제거 + `ai-docs/templates/{templateKey}/` 폴더 전체 삭제 + `pending-matches.md` 관련 행 삭제

## 7. Pruning 정책

케이스 예시는 **최대 10개**가 최적입니다. 11개 이상이 되면 AI 응답 속도 저하와 토큰(AI가 한 번에 처리하는 텍스트 양) 비용이 증가합니다.

### 언제 발동하나

`/add-example`로 예시를 추가한 뒤, 총 케이스 수가 11개 이상이 되면 자동으로 pruning 절차가 시작됩니다.

### 선별 기준

Claude가 아래 기준으로 10개를 추천합니다:

| 기준          | 설명                                                                 |
| ------------- | -------------------------------------------------------------------- |
| **다양성**    | 증상 양상·중증도·시간 경과가 서로 다른 케이스 우선                   |
| **완성도**    | 4섹션(HPI·P.I template·History·P/E)이 모두 충실히 채워진 케이스 우선 |
| **중복 제거** | 내용이 거의 동일한 케이스는 1개만 유지                               |

### 처리 흐름

1. Claude가 유지할 10개 목록과 제외 이유를 사용자에게 제시
2. **승인 시**: 10개만 남기고 번호를 1부터 재정렬하여 파일 덮어쓰기
3. **거절 시**: 사용자가 직접 남길 케이스 번호를 지정 → 동일하게 처리

## 8. 메타 학습 메커니즘

### 무엇인가

예시를 추가하거나 상용구를 등록할 때마다, Claude가 그 예시를 읽고 **현재 프롬프트 파일에 반영되지 않은 새로운 패턴이 있는지** 자동으로 분석합니다.

### 분석 대상 파일

- `ai-docs/prompts/generate-pi.md`
- `ai-docs/prompts/generate-template.md`
- `ai-docs/prompts/generate-history.md`
- `ai-docs/prompts/generate-pe.md`

### 흐름

```
새 예시 저장
    ↓
Claude가 4개 프롬프트 파일 검토
    ↓
새 패턴 발견?
  ├── 예 → 변경안을 diff 형식으로 사용자에게 제시
  │        ├── 승인 → 파일 업데이트
  │        └── 거절 → 건너뜀
  └── 아니요 → "기존 프롬프트와 잘 일치합니다. 추가 변경 없음." 보고
```

### 왜 중요한가

프롬프트 파일은 **모든 C.C.의 AI 생성 품질에 영향을 줍니다**. 특정 C.C.의 예시에서 발견된 좋은 패턴(예: 특정 시간 표기 방식, 특정 증상의 서술 순서)이 프롬프트 파일에 반영되면, 다른 C.C.의 생성 품질도 함께 향상됩니다.

이 메커니즘 덕분에 예시를 추가할수록 시스템 전체가 점진적으로 개선됩니다.

## 9. 향후 확장 — draft/corrected pair 형식

현재 `examples.md`는 "이상적인 출력 결과" 하나만 저장합니다.

앞으로는 아래와 같은 **pair 형식**으로 확장할 가능성이 있습니다.

```
## HPI (draft)         ← AI가 처음 생성한 초안
(AI 생성 결과)

## HPI (corrected)     ← 의사가 직접 수정한 최종본
(수정된 결과)
```

이 구조가 되면 AI가 "여기서 어떻게 수정했는지"까지 학습할 수 있어, 초안 품질이 더 빠르게 향상됩니다.

**지금 당장 바뀌는 건 없습니다.** 현재의 `examples.md` 형식은 이 확장을 염두에 두고 설계되어 있습니다. `loadExamples()` 함수의 섹션 이름 매칭이 prefix 기반(시작 문자열 비교)으로 되어 있는 것도 이런 이유입니다.

## 10. 주의사항

### 익명화 필수

`examples.md`에 저장하는 케이스 예시에는 **실제 환자 정보를 절대 포함하면 안 됩니다**. 나이·성별·날짜는 임의 값으로 바꾸고, 이름·주민번호·병원 코드 등 식별 가능한 정보는 모두 제거해야 합니다.

이 파일들은 Git에 저장되고 개발팀 전원이 볼 수 있습니다.

### 원본 카드 수정 금지

의사가 입력한 원본 카드 텍스트(`case_inputs.raw_text`)는 절대 수정하지 않습니다. AI 생성 결과가 마음에 들지 않아도, 원본 카드를 고치는 대신 생성된 결과를 편집창에서 직접 수정하세요.

### 파일 삭제 시 연동 체크

상용구나 가이드라인 파일을 삭제할 때는 반드시 아래 항목을 함께 정리해야 합니다:

- `cc-list.json`에서 해당 key 제거
- `template-list.json` 또는 `guide-list.json`에서 해당 항목 제거
- `pending-matches.md`에서 관련 행 삭제

이 중 하나라도 빠뜨리면 드롭다운에 없는 항목이 연결된 것처럼 보이거나, 삭제된 파일을 읽으려다 오류가 발생할 수 있습니다.

### `_generic` 스키마 주의

`ai-docs/templates/_generic/schema.json`은 등록된 스키마가 없는 C.C.에 자동으로 사용되는 범용 스키마입니다. 이 파일을 수정하면 **상용구가 없는 모든 C.C.의 1단계 정규화 결과**가 영향을 받습니다.
